# Техническое задание

## Проект: Ticket Tracker / Трекер заявок

## 1. Общая идея проекта

Нужно разработать веб-приложение для учета заявок.

Пользователь может:

* создавать заявки,
* просматривать свои заявки,
* редактировать часть данных заявки,
* оставлять комментарии,
* отслеживать изменение статуса.

Администратор может:

* просматривать все заявки,
* менять статус и приоритет,
* назначать исполнителя,
* оставлять комментарии,
* редактировать заявки,
* закрывать заявки.

Система должна иметь:

* backend на FastAPI,
* базу данных,
* простой web-интерфейс,
* авторизацию,
* разграничение прав доступа,
* валидацию данных,
* фильтрацию и поиск.

---

# 2. Цель проекта

Сделать MVP системы заявок, похожей на упрощённый helpdesk / issue tracker.

Проект должен показать, что ты умеешь:

* строить backend-приложение с несколькими сущностями,
* работать с CRUD,
* делать связи между таблицами,
* реализовывать авторизацию,
* ограничивать доступ по ролям,
* валидировать входные данные,
* делать фильтры, поиск и пагинацию,
* связывать backend с простым frontend.

---

# 3. Формат проекта

Приложение должно быть веб-приложением с двумя частями:

## Backend

Отвечает за:

* API,
* бизнес-логику,
* работу с БД,
* авторизацию,
* валидацию.

## Frontend

Простой интерфейс для работы с системой.

Фронтенд должен быть минималистичным, но рабочим.
Он нужен не для красоты, а чтобы:

* вручную создавать заявки,
* видеть список,
* открывать карточку заявки,
* менять статус,
* добавлять комментарии.

---

# 4. Роли пользователей

В системе должно быть **2 роли**.

## 4.1. User

Обычный пользователь.

Может:

* зарегистрироваться,
* войти в систему,
* создать заявку,
* видеть только свои заявки,
* открыть свою заявку,
* редактировать свою заявку, пока она не закрыта,
* оставлять комментарии в своих заявках,
* просматривать комментарии в своих заявках.

Не может:

* видеть чужие заявки,
* назначать исполнителя,
* менять статус на служебные статусы администратора,
* менять приоритет,
* удалять чужие данные.

---

## 4.2. Admin

Администратор / оператор системы.

Может:

* видеть все заявки,
* открывать любую заявку,
* менять статус,
* менять приоритет,
* назначать исполнителя,
* редактировать заявку,
* оставлять комментарии,
* закрывать и переоткрывать заявки.

---

# 5. Сущности системы

В проекте должны быть минимум следующие сущности:

1. User
2. Ticket
3. Comment
4. Status history

---

# 6. Сущность User

## Назначение

Хранит информацию о пользователях системы.

## Поля

* `id` — уникальный идентификатор
* `email` — уникальный email
* `username` — уникальное имя пользователя
* `hashed_password` — пароль в зашифрованном виде
* `role` — роль пользователя (`user` / `admin`)
* `is_active` — активен ли аккаунт
* `created_at` — дата создания
* `updated_at` — дата обновления

## Требования

* email должен быть уникальным
* username должен быть уникальным
* пароль в базе в открытом виде храниться не должен
* пользователь по умолчанию создается с ролью `user`

---

# 7. Сущность Ticket

## Назначение

Основная сущность проекта. Представляет заявку.

## Поля

* `id` — уникальный идентификатор
* `title` — краткий заголовок заявки
* `description` — подробное описание
* `status` — текущий статус
* `priority` — приоритет
* `author_id` — кто создал заявку
* `assignee_id` — кто назначен исполнителем, может быть пустым
* `created_at` — дата создания
* `updated_at` — дата обновления
* `closed_at` — дата закрытия, может быть пустой

## Допустимые значения `status`

Нужно использовать фиксированный набор статусов:

* `new`
* `in_progress`
* `resolved`
* `closed`

## Допустимые значения `priority`

Нужно использовать фиксированный набор приоритетов:

* `low`
* `medium`
* `high`

## Требования

* у каждой заявки обязательно должен быть автор
* статус по умолчанию при создании: `new`
* приоритет по умолчанию при создании: `medium`
* `closed_at` заполняется только когда заявка переводится в `closed`
* если заявка переоткрыта, `closed_at` должен очищаться

---

# 8. Сущность Comment

## Назначение

Комментарии внутри заявки.

## Поля

* `id` — уникальный идентификатор
* `ticket_id` — заявка, к которой относится комментарий
* `author_id` — автор комментария
* `content` — текст комментария
* `created_at` — дата создания
* `updated_at` — дата обновления

## Требования

* комментарий всегда привязан к конкретной заявке
* комментарий всегда имеет автора
* пустой комментарий запрещён

---

# 9. Сущность StatusHistory

## Назначение

Хранит историю изменения статусов заявки.

## Поля

* `id` — уникальный идентификатор
* `ticket_id` — ссылка на заявку
* `changed_by` — кто изменил статус
* `old_status` — старый статус
* `new_status` — новый статус
* `changed_at` — дата изменения

## Требования

* запись создается только при изменении статуса
* история должна сохраняться даже если заявку потом снова изменят
* историю нельзя редактировать вручную через интерфейс

---

# 10. Связи между сущностями

## User → Ticket

* один пользователь может создать много заявок
* одна заявка имеет одного автора

## User → Ticket (assignee)

* один администратор может быть назначен на много заявок
* у заявки может быть один исполнитель или никто

## Ticket → Comment

* одна заявка может иметь много комментариев
* один комментарий относится только к одной заявке

## Ticket → StatusHistory

* одна заявка может иметь много записей истории статусов
* одна запись истории относится только к одной заявке

---

# 11. Бизнес-правила

Это самый важный раздел.

## 11.1. Создание заявки

Обычный пользователь может создать заявку, указав:

* заголовок
* описание

При создании:

* статус автоматически = `new`
* приоритет автоматически = `medium`
* исполнитель не назначается

---

## 11.2. Просмотр заявок

### User

Видит только:

* заявки, созданные им самим

### Admin

Видит:

* все заявки в системе

---

## 11.3. Изменение заявки

### User может:

* изменить `title`
* изменить `description`

Но только если:

* заявка принадлежит ему
* заявка не находится в статусе `closed`

### User не может:

* менять статус
* менять приоритет
* назначать исполнителя

### Admin может:

* менять `title`
* менять `description`
* менять `status`
* менять `priority`
* назначать `assignee`

---

## 11.4. Изменение статуса

Изменять статус может только администратор.

Разрешенные переходы:

* `new` → `in_progress`

* `new` → `resolved`

* `new` → `closed`

* `in_progress` → `resolved`

* `in_progress` → `closed`

* `resolved` → `closed`

* `resolved` → `in_progress`

* `closed` → `in_progress`

Нельзя:

* переводить заявку в тот же статус без изменения
* устанавливать статус вне списка допустимых значений

При каждом изменении статуса:

* обновляется поле `updated_at`
* создается запись в `StatusHistory`

---

## 11.5. Закрытие заявки

Когда статус становится `closed`:

* поле `closed_at` должно быть заполнено текущей датой и временем

Если заявка выводится из `closed` в `in_progress`:

* поле `closed_at` должно стать пустым

---

## 11.6. Назначение исполнителя

Только администратор может назначить исполнителя.

Правила:

* исполнителем может быть только пользователь с ролью `admin`
* исполнитель может быть не назначен
* замена исполнителя разрешена

---

## 11.7. Комментарии

### User

Может добавлять комментарии:

* только в свои заявки

### Admin

Может добавлять комментарии:

* в любую заявку

Редактирование комментариев:

* **в MVP запрещено**
* комментарии после создания не изменяются

Удаление комментариев:

* **в MVP запрещено**

---

## 11.8. Удаление заявок

В MVP:

* физическое удаление заявок запрещено

Причина:

* не ломать историю и комментарии

---

# 12. Валидация данных

## 12.1. Регистрация

### Email

* обязателен
* должен быть корректного формата
* должен быть уникальным

### Username

* обязателен
* длина: от 3 до 30 символов
* уникален
* только буквы, цифры, `_`, `-`

### Password

* обязателен
* длина: минимум 8 символов

---

## 12.2. Ticket.title

* обязателен
* длина: от 5 до 100 символов
* не должен состоять только из пробелов

## 12.3. Ticket.description

* обязателен
* длина: от 10 до 2000 символов
* не должен состоять только из пробелов

## 12.4. Comment.content

* обязателен
* длина: от 1 до 1000 символов
* не должен состоять только из пробелов

* защищенные маршруты недоступны без авторизации

### Только авторизованные пользователи:

* просмотр своих данных
* создание заявок
* комментарии
* просмотр списка заявок

### Только admin:

* просмотр всех заявок
* изменение статусов
* изменение приоритета
* назначение исполнителя

Это не про красоту, а про базовую удобность.

Должно быть:

* понятное отображение статуса
* понятное отображение приоритета
* сообщения об ошибках в форме
* сообщение об успешном создании / обновлении

# 19. Ошибки и ответы системы

Система должна корректно обрабатывать ошибки.

Примеры ситуаций:

* неверный логин или пароль
* попытка открыть несуществующую заявку
* попытка получить доступ к чужой заявке
* попытка user изменить статус
* попытка назначить не-admin пользователя исполнителем
* отправка невалидных данных

Нужно возвращать:

* корректный HTTP status code
* понятное сообщение об ошибке

# 20. Нефункциональные требования

## Код

* проект должен быть структурирован
* код должен быть читаемым
* не должно быть всей логики в одном файле

## База данных

* все данные должны храниться в БД
* данные между перезапусками не должны теряться

## Безопасность

* пароль нельзя хранить в открытом виде
* закрытые маршруты должны быть защищены
* обычный пользователь не должен получать данные чужих заявок

## Логика

* сервер не должен позволять действия, которые запрещены по ролям, даже если пользователь вручную отправил запрос

# 22. Что НЕ входит в MVP

Чтобы не перегрузить первый проект, в MVP **не включать**:

* загрузку файлов
* удаление комментариев
* редактирование комментариев
* email-уведомления
* websocket
* чат
* soft delete для всех сущностей
* сложную аналитику
* dashboard с графиками
* несколько уровней ролей
* подтверждение email
* reset password по почте

# 25. Основные пользовательские сценарии

## Сценарий 1

Обычный пользователь:

* регистрируется
* входит в систему
* создает заявку
* открывает свою заявку
* пишет комментарий
* видит, что админ изменил статус

---

## Сценарий 2

Админ:

* входит в систему
* открывает список всех заявок
* фильтрует заявки по статусу `new`
* открывает заявку
* назначает себя исполнителем
* меняет статус на `in_progress`
* пишет комментарий
* позже переводит заявку в `resolved`
* затем в `closed`

---

## Сценарий 3

Обычный пользователь:

* пытается открыть чужую заявку
* получает ошибку доступа
